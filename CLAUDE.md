# LLM Wiki — Schema

你是本 Wiki 的维护者。你的职责是按照以下规范，持续将 `raw/` 中的原始素材编译为结构化、交叉引用的 Markdown 知识库。你负责所有写入操作，用户负责素材收集、方向引导和质量把关。

---

## 三层架构

### 第一层：索引层

`wiki/index.md` 是全站内容目录。每个 Wiki 页面在此占一行，按分类组织。LLM 每次查询时**先读 index.md 定位相关页面**，再深入阅读具体页面。每次 ingest 后必须同步更新 index.md。

索引条目格式：
```markdown
- [页面标题](相对路径) — 一句话摘要 `#标签1 #标签2`
```

分类区块：
- **实体**：人物、模型、工具、数据集、公司/组织
- **概念**：技术概念、方法论、理论
- **论文**：学术论文摘要
- **主题**：技术领域、研究方向、行业动态
- **综述**：多源对比、综合分析、阶段性总结

### 第二层：摘要层

每个原始素材（论文、文章、视频等）对应一个摘要页面，放在 `wiki/papers/` 或 `wiki/topics/` 下。摘要页面包含：
- 来源信息（作者、日期、URL）
- 核心观点（3-8 条要点）
- 关键引用或数据
- 指向相关实体/概念页面的链接
- 指向详细实体页面的"延伸阅读"链接

摘要长度：300-800 字。

### 第三层：详细实体层

实体页面 (`wiki/entities/`)、概念页面 (`wiki/concepts/`)、主题页面 (`wiki/topics/`) 和综述页面 (`wiki/synthesis/`) 构成知识的主体。

- **实体页**：描述一个人物、模型、工具、数据集等的完整信息，汇总与该实体相关的所有摘要引用
- **概念页**：深入解释一个技术概念，包含定义、原理、变体、相关论文、代码示例
- **主题页**：覆盖一个技术领域或研究方向的全景视图
- **综述页**：跨多个来源的对比分析、趋势总结、阶段性反思

这些页面由 LLM 在 ingest 过程中持续更新——新素材的信息被合并进已有页面，而非简单追加。

## 目录结构

```
llm-wiki/
├── CLAUDE.md              # 本文件 — Wiki 操作规范
├── raw/                   # 原始素材（不可变，仅追加）
│   └── assets/            # 图片等附件
├── wiki/                  # LLM 维护的所有 Markdown
│   ├── index.md           # 全站索引
│   ├── log.md             # 操作日志
│   ├── entities/          # 实体：人物、模型、工具、数据集
│   ├── concepts/          # 概念：技术概念、方法论
│   ├── papers/            # 论文摘要
│   ├── topics/            # 主题/领域/技术方向
│   └── synthesis/         # 综述/对比/阶段总结
└── templates/             # 页面模板（参考用）
```

## 页面命名规范

- 文件名使用英文小写 + 连字符：`transformer-architecture.md`、`andrew-ng.md`
- 页面标题（H1）使用中文：`# Transformer 架构`
- 论文摘要页以论文简称命名：`attention-is-all-you-need.md`
- 人名为姓-名格式：`geoffrey-hinton.md`，标题用全名 `# Geoffrey Hinton`
- 避免特殊字符和空格

## 工作流：Ingest（摄入）

当用户要求摄入新素材时，按以下步骤操作：

1. **读取素材**：读取 `raw/` 中的源文件。如果是网页文章，用 Obsidian Web Clipper 转换为 Markdown 后放入 `raw/`。
2. **讨论关键点**：与用户简要讨论素材的核心内容和值得强调的部分。
3. **创建/更新摘要页**：在 `wiki/papers/` 或 `wiki/topics/` 下创建摘要页面，遵循对应模板。
4. **更新实体页**：遍历摘要中涉及的所有实体（模型、人物、工具、数据集），如果实体页不存在则创建，存在则更新——在"相关摘要"区块添加指向新摘要的链接，如果新素材带来了新信息则更新正文。
5. **更新概念页**：同理，如果素材涉及某个概念的新角度或深入解释，更新对应的概念页。
6. **检查交叉引用**：确保新页面和更新的页面中，相关实体和概念的链接都正确。
7. **更新 index.md**：在所有涉及的分类区块中添加或更新条目。
8. **追加 log.md**：按时间格式记录本次操作。
9. **报告**：简要告诉用户本次 ingest 触达了哪些页面。

## 工作流：Query（查询）

当用户提问时，按以下步骤操作：

1. **读取 index.md**：找出所有可能相关的页面。
2. **读取相关页面**：按照索引中的链接，读取相关的摘要页、实体页、概念页。
3. **合成答案**：基于读取的内容给出结构化回答，附带引用来源（链接到 Wiki 页面）。
4. **归档（可选）**：如果用户的问题产生了有价值的分析或新视角，主动询问是否将答案归档为 `wiki/synthesis/` 下的新页面。

## 工作流：Lint（体检）

用户可以随时要求进行 Wiki 健康检查。检查项目：

1. **矛盾检测**：搜索不同页面对同一事实的不同描述，标注矛盾。
2. **过时信息**：检查是否有较新的素材推翻或修正了旧页面的论断。
3. **孤立页面**：通过反向链接检查是否有页面没有任何其他页面引用它（log.md 和 index.md 不算）。
4. **缺失页面**：检查 index.md 中提到的实体/概念是否都有对应页面。
5. **断链排查**：检查所有 `[[wikilink]]` 和相对链接是否指向存在的文件。
6. **内容缺口**：基于现有知识结构，建议可以进一步研究的方向或应添加的素材。
7. **格式合规**：检查页面 frontmatter 是否完整、模板结构是否正确。

输出 lint 报告，标注严重程度（严重 / 建议 / 提示），并询问用户是否立即修复。

## index.md 维护规范

- 每次 ingest 后更新
- 每次创建新页面后更新
- 条目按分类组织，同一分类内按创建时间排序（新的在前）
- 每个条目一行，包含：链接 + 一句话摘要 + 标签
- 一句话摘要要能独立理解，不可仅写"关于 XX 的文章"——必须写出核心观点

示例：
```markdown
## 实体
- [GPT-4](wiki/entities/gpt-4.md) — OpenAI 2023年发布的多模态大语言模型 `#模型 #OpenAI`

## 概念
- [RLHF](wiki/concepts/rlhf.md) — 基于人类反馈的强化学习对齐方法 `#对齐 #训练方法`

## 论文
- [Attention Is All You Need](wiki/papers/attention-is-all-you-need.md) — Transformer 架构的开创性论文，提出了自注意力机制取代循环神经网络 `#transformer #注意力机制`
```

## log.md 维护规范

- 纯追加，不修改历史条目
- 每次 ingest / query / lint 后追加一行
- 格式：`## [YYYY-MM-DD] <操作类型> | <简述>`
- 操作类型：`ingest`、`query`、`lint`、`update`
- 每个条目下可附 2-3 条要点（无序号列表）

示例：
```markdown
## [2026-07-04] ingest | Attention Is All You Need 论文
- 创建论文摘要页
- 更新 Transformer 概念页
- 创建 Self-Attention 概念页
```

## 语言与风格

- **Wiki 正文优先使用中文**，专业术语在首次出现时标注英文原名，如"注意力机制（Attention Mechanism）"
- **文件名、标签、frontmatter 字段名使用英文**
- **代码块、公式、论文标题保留原文**
- 写作风格：简洁、客观、结构清晰。避免感叹号和主观评价词（"令人惊叹""颠覆性"等）
- 外部引用使用 Markdown 链接 `[文本](url)`，内部引用使用 Obsidian 兼容的 `[[页面名]]`

---

## 质量原则

1. **一个页面说清一件事**：实体页聚焦该实体本身，不嵌套大量概念解释（用链接解决）
2. **摘要要真正"摘要"**：不是原文翻译，而是提炼出对已有知识库的增量贡献
3. **交叉引用比内容多更重要**：宁可少写一段，不可少一个链接
4. **矛盾优先标注**：发现新素材与已有页面冲突时，标注而非覆盖，让用户判断
5. **保持谦逊**：不确定的内容标注"有待查证"或"（据论文 X 声称）"
